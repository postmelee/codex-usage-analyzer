# Task M040 #48 수행계획서

GitHub Issue: [#48](https://github.com/postmelee/codex-usage-analyzer/issues/48)
마일스톤: M040

## 목적

PR #47로 `main`에 병합된 opt-in `profile` command와 Experimental Full Profile Envelope를 npm 사용자에게 제공하는 `0.4.0` minor release를 준비하고 실행한다. Feature 구현을 다시 변경하지 않고 package, CLI, SDK declaration, stable/experimental app-server client와 test assertion의 version surface만 `0.4.0`으로 일치시킨다.

Release preparation PR merge 후 최신 `main` merge commit에 `v0.4.0` tag를 만들고, 기존 GitHub Actions `Publish Package` workflow와 npm Trusted Publisher OIDC를 사용한다. Registry, published `npx`, package/README, ECDSA signature와 provenance를 검증한 뒤 마지막으로 GitHub Release를 만들고 Issue #48을 종료한다.

## 배경

Issue #46과 PR #47은 다음 M040 기능을 구현·검증했다.

- 기본 인자 없음, `usage`, `--json`, `usage --json`은 공식 `account/usage/read`와 identity-free Account Usage Contract v1을 유지한다.
- 명시적 `profile`과 `profile --json`은 unsupported private endpoint를 사용하는 별도 experimental 경로다.
- Full Profile Envelope v1/schema, dedicated transport, human renderer와 52주 token activity map을 추가했다.
- Credential은 dedicated child/request 수명에 제한하고 fixed HTTPS target, manual redirect, no retry, timeout과 response limit을 적용했다.
- Public SDK와 Account Usage Contract v1은 변경하지 않았다.
- 최종 regression 116개, npm artifact 23개, runtime dependency 0개와 승인된 live structural parity를 검증했다.

현재 source와 release baseline은 다음과 같다.

- Main package/runtime version: `0.3.0`
- npm registry `latest`: `0.3.0`
- Existing Git tag/GitHub Release: `v0.3.0`
- Full regression: 116 tests
- Package dry-run: 23 files, 약 29.1 kB
- `package.json` files allowlist: 16 paths
- Runtime/development dependency: 0개
- Lockfile: 없음
- Local runtime: Node `v24.15.0`, npm `11.12.1`
- Open Dependabot PR #40/#41: CI와 publish workflow의 `actions/setup-node`/`actions/checkout` major update

`0.4.0`은 기존 stable CLI/SDK/Account Usage Contract를 깨지 않는 명시적 experimental command 추가이므로 repository release guide의 minor 기준에 해당한다. `0.4.0` publish 전에는 `npx codex-usage-analyzer@latest profile`이 제공되지 않는다.

Release 운영의 진실 원천은 [`npm_release_guide.md`](../manual/npm_release_guide.md)다. Trusted Publishing은 `contents: read`, `id-token: write`, GitHub-hosted runner와 short-lived OIDC credential을 사용한다. Long-lived npm token과 수동 `--provenance` flag를 추가하지 않는다.

## 설계 방향

### Version surface

- `npm version --no-git-tag-version 0.4.0`으로 `package.json`만 기계적으로 bump하고 tag는 만들지 않는다.
- Lockfile이 없으므로 신규 lockfile을 생성하지 않는다.
- Source version 위치는 다음 5개로 제한한다.
  - `package.json`
  - `src/cli.js`
  - `src/index.d.ts`
  - `src/app-server-client.js`
  - `src/experimental-profile-client.js`
- Test assertion은 다음 4개 파일에서만 갱신한다.
  - `src/__tests__/cli.test.js`
  - `src/__tests__/index.test.js`
  - `src/__tests__/app-server-client.test.js`
  - `src/__tests__/experimental-profile-client.test.js`
- Global replacement를 사용하지 않는다. Benchmark와 과거 release/task 문서의 historical `0.2.0`/`0.3.0`은 변경하지 않는다.
- `src/index.js`, Account Usage/Full Profile normalizer·schema와 renderer는 변경하지 않는다.

### Package and documentation boundary

- README와 `docs/`는 #46에서 사용자/consumer 설명을 완료했으므로 #48에서 수정하지 않는다.
- Package artifact에는 최소 다음 항목이 포함돼야 한다.
  - Stable Account Usage runtime와 contract/schema
  - Experimental profile client, normalizer와 renderer
  - Full Profile contract/schema
  - README와 downstream integration guide
- Package artifact에서 tests, `.github/`, `mydocs/`, scripts, local/extracted 분석물과 auth/raw-response fixture를 제외한다.
- npm publish 후 registry README에서 `profile`, `profile --json`, unsupported `/wham/profiles/me`, official `account/usage/read`와 Full Profile contract/schema link marker를 확인한다.
- Release note에는 stable default 무변경, explicit experimental profile, identity/privacy와 unsupported endpoint 제한을 요약하되 실제 계정 값은 포함하지 않는다.

### Release ordering and approval

- Pre-merge Stage에서는 version source/test만 변경한다. Tag, workflow dispatch, npm publish와 GitHub Release를 수행하지 않는다.
- PR body에는 Issue #48 closing keyword를 넣지 않는다.
- PR merge 후에도 Issue #48과 M040은 release 완료 전까지 열린 상태를 유지한다.
- `pr-merge-cleanup`은 post-release gate 완료와 Issue close 시점까지 보류한다.
- Tag 생성, tag push, strict preflight, workflow dispatch, npm publish와 GitHub Release는 PR merge 후 작업지시자의 별도 명시 승인을 받아 수행한다.
- `v0.4.0` tag는 release preparation PR의 최신 `main` merge commit을 가리켜야 한다.
- Tag push 후 `npm run release:preflight -- --release-ready`가 전부 OK일 때만 publish workflow를 dispatch한다.
- `gh workflow run publish.yml --ref v0.4.0`으로 immutable tag ref를 실행한다.
- Publish 성공 후 registry/npx/package/README/signature/provenance를 검증한 다음 마지막으로 GitHub Release를 생성한다.
- Publish 후 검증이 실패해도 `0.4.0`을 재게시하지 않는다. Issue를 partial-release 상태로 유지하고 후속 patch를 결정한다.

### Supply-chain boundary

- Registry metadata에서는 signature/provenance 존재와 predicate category만 기록하고 raw signature/attestation을 task 문서에 복사하지 않는다.

## 리스크

- **Irreversible npm version**: npm publish 후 같은 version을 재사용할 수 없다. Strict preflight와 artifact audit 전 workflow를 실행하지 않는다.
- **External partial release**: Publish 후 검증이 실패하면 재게시하지 않고 Issue를 열린 상태로 유지해 후속 patch를 결정한다.
- **Premature mutation**: PR merge와 별도 작업지시자 승인 전 tag, workflow dispatch, publish, Release를 만들지 않는다.
- **Premature cleanup**: PR merge 직후 issue close/cleanup을 수행하지 않고 post-release 검증 완료까지 Task #48을 유지한다.
- **Version over-replacement**: Exact source/test allowlist만 수정하고 historical benchmark/release/task 문서 marker를 보존한다.
- **Stable contract regression**: Public SDK, Account Usage와 Full Profile runtime/schema를 diff-zero와 full regression으로 보호한다.
- **Private API drift**: 이번 release는 endpoint를 수정하지 않는다. Postpublish는 account access 없는 `profile --help`를 기본 smoke로 사용하고 실제 profile 호출은 별도 승인 없이는 수행하지 않는다.
- **Trusted Publisher mismatch**: Existing successful v0.3.0 workflow/OIDC boundary를 보존하고 long-lived credential을 추가하지 않는다.
- **Signature/provenance drift**: Registry metadata와 `npm audit signatures`를 함께 검증하고 raw material을 기록하지 않는다.
- **Dependabot conflict**: PR #40/#41이 먼저 merge되면 최신 workflow를 보존하되 action update를 #48 diff에 섞지 않는다.
- **Registry propagation delay**: Publish 직후 bounded retry 후에도 metadata/README/npx가 불일치하면 partial-release 상태로 중단한다.

## 승인 요청 사항

- M040 기능을 backward-compatible minor release `0.4.0`으로 확정하는 방향
- Version source/test 9개 파일만 수정하고 feature/runtime contract와 public docs를 변경하지 않는 범위
- 3개 pre-merge Stage와 별도 post-merge release run으로 분할하는 절차
- PR #40/#41을 release 범위에서 제외하고 tag 시점의 최신 main workflow만 재검증하는 정책
- PR에 closing keyword를 넣지 않고 post-release 검증 후 Issue #48과 cleanup을 완료하는 순서
- PR merge 후 tag/publish/GitHub Release 전에 별도 작업지시자 승인을 요청하는 안전 경계
