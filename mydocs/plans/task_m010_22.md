# Task M010 #22 수행계획서

GitHub Issue: [#22](https://github.com/postmelee/codex-usage-analyzer/issues/22)
마일스톤: M010

## 목적

`codex-usage-analyzer`의 npm release 보안 경로를 manual publish 중심에서 GitHub Actions OIDC 기반 trusted publishing과 provenance 검증 가능한 구조로 옮길지 결정한다. 도입하기로 한 경우 저장소에서 관리 가능한 최소 workflow와 README release 안내를 설계 또는 구현하고, npmjs.com package setting처럼 저장소 밖에서 필요한 maintainer 조치를 명확히 남긴다.

이번 task는 신규 npm version publish가 아니라 release 보안 구조를 확정하는 작업이다. long-lived npm token, npm account identifier, credential이 문서나 PR 본문에 노출되지 않도록 한다.

## 배경

#7에서 `codex-usage-analyzer@0.1.0`을 manual publish로 배포했고 postpublish `npx @latest` smoke까지 확인했다. #21에서 license metadata도 MIT로 확정했다. 남은 release 운영 리스크는 npm publish 인증 방식과 provenance 검증 가능성이다.

2026-07-03 기준 npm 공식 문서 확인 결과:

- trusted publishing은 OIDC 기반 CI/CD publish로 long-lived npm token 없이 publish할 수 있는 경로다.
- npm trusted publishing은 npm CLI `11.5.1` 이상과 Node `22.14.0` 이상을 요구한다.
- GitHub Actions 사용 시 GitHub-hosted runner, workflow `permissions.id-token: write`, `contents: read`, `actions/setup-node`의 registry 설정이 핵심이다.
- GitHub Actions 또는 GitLab CI/CD trusted publishing으로 publish하면 provenance attestation은 기본으로 생성되며 `--provenance` flag가 필요 없다.
- token 기반 provenance만 사용할 경우 GitHub Actions의 `id-token: write`, GitHub-hosted runner, `npm publish --provenance`가 필요하다.
- package provenance는 악성 코드 부재를 보장하지 않고, package가 어디서 어떻게 build/publish 되었는지 검증 가능한 연결을 제공한다.

## 설계 방향

- 기본 선호 방향은 trusted publishing 채택이다. long-lived npm token을 제거하고 provenance를 자동 생성할 수 있기 때문이다.
- 저장소에서 구현 가능한 항목과 npmjs.com에서 maintainer가 수행해야 하는 항목을 분리한다.
- release workflow를 추가하는 경우 GitHub-hosted `ubuntu-latest`, Node `24`, `permissions: id-token: write`, `contents: read`, `actions/setup-node` registry 설정, `package-manager-cache: false`를 기본값으로 검토한다.
- workflow trigger는 #23 version bump/release 절차와 충돌하지 않도록 보수적으로 설계한다. 후보는 `workflow_dispatch` 중심 초안, `release.published`, `push tags: v*` 중 Stage 1에서 비교한다.
- 이번 task에서는 publish를 실행하지 않는다. workflow가 추가되더라도 실제 publish는 trusted publisher 설정과 별도 release approval이 충족된 뒤 후속 release에서 수행한다.
- provenance 검증은 `npm audit signatures`를 release 후 소비자/maintainer 검증 수단으로 문서화할지 검토한다.
- 기존 CI는 package test/dry-run smoke 용도로 유지하고, publish workflow는 release 전용으로 분리하는 방향을 우선 검토한다.

- 도입하지 않는 경우 manual publish 유지 이유와 재검토 조건이 기록된다.

- 이번 task에서 신규 npm publish를 실행하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.

## 리스크

- **npmjs.com 설정은 저장소 밖 상태**: trusted publisher 등록은 npm package settings에서 수행해야 하므로, 이번 task에서는 필요한 설정값과 maintainer action을 문서화하고 source workflow만 리뷰 가능한 상태로 만든다.
- **runtime 조건 불일치**: 현재 CI는 Node 20이지만 trusted publishing은 Node 22.14.0+와 npm CLI 11.5.1+ 조건이 있으므로 release workflow는 별도 Node 버전을 사용해야 할 수 있다.
- **실수 publish 위험**: 이번 task에서는 publish를 실행하지 않고, workflow trigger도 Stage 1에서 보수적으로 검토한다.
- **token 노출 위험**: trusted publishing 경로에서는 npm token을 저장하지 않는 방향을 우선하며, token/provenance 대안 검토 시에도 secret value나 account identifier를 문서에 남기지 않는다.
- **#23과의 절차 중복**: version bump, tag, GitHub Release 전체 절차는 #23으로 남기고, #22는 trusted publishing/provenance 보안 경로에 집중한다.
