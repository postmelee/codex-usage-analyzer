# Task M010 #15 수행계획서

GitHub Issue: [#15](https://github.com/postmelee/codex-usage-analyzer/issues/15)
마일스톤: M010

## 목적

Codex Desktop profile UI와 `codex-usage-analyzer` local analyzer 출력의 source 차이를 줄일 수 있는지 판단하기 위해 Desktop profile/cache/source 후보를 inventory한다.

이번 task의 결과는 parser 구현이 아니라 source decision이다. local에서 안전하게 읽을 수 있는 source, 읽을 수 없거나 사용하면 안 되는 source, privacy 위험이 큰 source를 구분하고, #16 analyzer local source coverage 확장에서 실제 구현할 수 있는 범위를 명확히 넘긴다.

## 배경

#6은 redacted profile baseline smoke 절차를 추가했고, #14는 remote/local source 차이를 `source_mismatch`와 `profile_parity_not_guaranteed` reason으로 더 명확히 표현했다. 두 작업 모두 Codex Desktop profile 값이 remote account-level source일 수 있고, 현재 analyzer는 local session JSONL과 제한된 local metadata만 읽는다는 한계를 남겼다.

현재 README는 production `analyze --json` 경로가 local Codex session JSONL을 읽고, Desktop profile parity는 보장하지 않는다고 설명한다. #15는 이 진단을 한 단계 더 구체화해 Desktop app data/cache나 공개 Codex source/reference에서 profile parity를 높일 수 있는 local source가 있는지 조사한다.

## 범위

### 포함

- Codex Desktop local app data/cache 후보 inventory
- 기존 analyzer parser source와 diagnostics contract 재검토
- #6, #14 profile smoke 결과와 README source/parity 설명 재검토
- 기존 `codex-extracted` 분석 결과와 공개 Codex repo/reference 재검토
- profile UI의 total/streak/top plugins/activity insight 관련 cache 후보 분류
- privacy-safe read policy와 문서 redaction 규칙 정리
- #16 parser 구현으로 넘길 수 있는 source와 제외해야 할 source 결정

### 제외

- Codex Desktop remote/internal profile API 호출
- 인증 토큰, 쿠키, session credential 추출 또는 사용
- Codex Desktop UI 자동 조작, screenshot OCR, profile page scraping
- parser 구현 또는 `UsageSnapshot v2` schema 변경
- 실제 사용자 raw profile 값, raw analyzer JSON, local absolute path, account identifier, credential 저장
- npm release, publish workflow, version bump 관련 작업

## 설계 방향

- 조사 결과는 `mydocs/working/` 단계 보고서와 `mydocs/report/` 최종 보고서에만 남긴다. 사용자-facing README나 analyzer code는 #15에서 수정하지 않는다.
- local source 후보는 실제 개인 경로 대신 `$HOME`, `{codex-home}`, `{app-support}` 같은 placeholder로 기록한다.
- 실제 파일 내용은 raw dump하지 않고, file kind, schema shape, field availability, privacy classification만 요약한다.
- 민감정보처럼 보이는 key/value, account id, email, token, absolute private path는 문서와 commit에 포함하지 않는다.
- source classification은 최소한 다음 네 가지로 구분한다.
  - `usable_local_source`: #16에서 parser 후보로 검토 가능
  - `unavailable_or_absent`: 현재 local 환경 또는 공개 reference에서 확인 불가
  - `remote_only_or_internal`: local analyzer가 사용하지 않아야 하는 source
  - `privacy_risk`: 읽기 또는 문서화가 위험해 제외해야 하는 source
- Desktop profile parity가 불가능하거나 부적절한 필드는 현재처럼 diagnostics와 profile smoke reason으로 표현하고, parser 구현으로 해결하지 않는다.

## 리스크

- **privacy leakage**: source inventory 과정에서 실제 사용자 데이터나 private path가 문서에 남을 수 있다. 문서에는 placeholder path와 schema summary만 기록하고, privacy scan을 단계 검증에 포함한다.
- **source ambiguity**: Desktop profile 값이 remote account-level source라면 local cache만으로 parity를 높일 수 없을 수 있다. 이 경우 `remote_only_or_internal`로 명시하고 parser 구현 범위에서 제외한다.
- **scope creep**: source 후보 발견 후 곧바로 parser를 구현하고 싶어질 수 있다. #15는 inventory와 feasibility decision으로 제한하고, 구현은 #16으로 넘긴다.
- **local environment bias**: 한 로컬 환경에서 후보가 없다고 일반적으로 없다고 단정할 수 없다. 공개 reference와 기존 산출물을 함께 검토하고, 확인 범위와 한계를 보고서에 적는다.
- **documentation churn**: README를 조사 결과로 바로 수정하면 사용자-facing 계약이 흔들릴 수 있다. #15에서는 공식 문서 변경을 제외하고 #16에서 필요성을 재판단한다.
