# Task M010 #30 수행계획서

GitHub Issue: [#30](https://github.com/postmelee/codex-usage-analyzer/issues/30)
마일스톤: M010

## 목적

#30의 목적은 Codex/ChatGPT credential을 사용하는 remote profile endpoint 조회를 기본 analyzer와 분리된 opt-in experimental command로 둘 수 있는지 판단하는 것이다.

이번 task는 remote command를 기본 기능으로 채택하는 구현 task가 아니다. local analyzer의 안전한 기본 동작을 유지하면서, remote profile-derived 값이 local analyzer보다 실제 Codex profile UI와 더 잘 맞는지 검증할 수 있는 조건, 보안 경계, 성능 측정 방법, 채택/폐기 기준을 문서화한다.

## 배경

#15에서 Desktop profile remote 값을 privacy-safe local cache에서 재사용하는 경로는 확인하지 못했다. 따라서 Codex Desktop profile UI와 local analyzer 결과가 계속 다를 가능성이 있다.

반면 `junhoyeo/tokscale`에는 local Codex session JSONL parser와 별도로 Codex auth credential을 읽어 ChatGPT backend usage/rate-limit API를 호출하는 경로가 있다. 이 방식은 기술적으로 참고할 수 있지만, npm 패키지의 기본 analyzer가 credential/keychain을 읽고 비공식 backend endpoint를 호출하는 순간 보안 신뢰 모델이 바뀐다.

따라서 #30은 remote endpoint 호출을 기본 analyzer에 넣는 것이 아니라, 별도 opt-in experimental command로 feasibility를 검증할 수 있는지 판단하는 작업으로 진행한다.

## 범위

### 포함

- `/api/codex/profiles/me`, `/wham/profiles/me` 계열 endpoint의 조사 가능성과 한계를 정리한다.
- opt-in experimental command 또는 별도 probe command의 CLI surface 후보를 정리한다.
- Codex auth file/keychain 접근이 필요한 경우의 사용자 동의, 로그, 저장 금지 정책을 정리한다.
- local analyzer output과 remote profile-derived output을 비교할 field, tolerance, source mismatch reason을 정리한다.
- latency와 실패 모드 측정 기준을 정리한다.
- 실제 credential이나 raw response 없이 검증 가능한 mock fixture/test 전략을 정리한다.
- 채택, 보류, 폐기 중 어느 결론이 맞는지 판단한다.

### 제외

- 기본 `analyze` 경로에 remote call을 추가하지 않는다.
- auth token, refresh token, account id, raw profile response, local absolute path를 저장소나 PR 본문에 남기지 않는다.
- npm publish 전 기본 동작을 변경하지 않는다.
- 비공식 endpoint를 안정 API로 전제하지 않는다.
- `UsageSnapshot v2` schema를 변경하지 않는다.
- 실제 사용자의 raw Codex profile 값이나 screenshot을 commit하지 않는다.

## 설계 방향

- 기본 analyzer는 local-only source policy를 유지한다.
- remote profile 조회는 가능하더라도 `experimental`, `opt-in`, `non-default`, `no persistence` 원칙을 따라야 한다.
- credential 접근이 필요한 단계는 구현계획서에서 별도 승인 항목으로 분리하고, 승인 전에는 실제 credential/keychain을 읽지 않는다.
- live remote probe가 필요하면 raw response를 저장하지 않고, redacted metric summary만 작업 문서에 남긴다.
- endpoint shape는 비공식으로 취급하며, schema drift와 auth failure가 정상 실패 모드로 처리되는 설계를 전제한다.
- mock fixture는 실제 endpoint raw response가 아니라 synthetic/redacted shape만 사용한다.

## 잠정 단계

- **Stage 1 — remote source와 credential boundary 조사**
  - 산출물: endpoint 후보, auth source 후보, 비공식 API 리스크, `tokscale` reference 차이를 정리한 Stage 1 보고서
  - 검증 관점: raw credential/raw response를 남기지 않고 source boundary가 문서화됐는지 확인한다.
- **Stage 2 — experimental command/probe 설계와 mock 검증 전략**
  - 산출물: opt-in CLI surface 후보, no-persistence guardrail, mock fixture/test 전략, failure handling matrix
  - 검증 관점: 기본 `analyze` 경로와 분리되는지, 실제 endpoint 없이 테스트 가능한지 확인한다.
- **Stage 3 — parity/latency 검증 프로토콜과 최종 feasibility 판단**
  - 산출물: local analyzer 대비 remote profile-derived 비교 기준, latency 측정 기준, 채택/보류/폐기 판단, 최종 보고서
  - 검증 관점: 실제 민감값 없이 결론을 재검토할 수 있고 후속 구현 범위가 분리되는지 확인한다.

## 리스크

- **Credential boundary 오해**: experimental command가 기본 analyzer처럼 보이면 npm 패키지 신뢰 모델이 훼손된다. 기본 경로와 별도 opt-in 경계를 문서와 설계에서 반복 확인한다.
- **비공식 endpoint drift**: endpoint shape나 auth 방식이 예고 없이 바뀔 수 있다. 안정 API로 전제하지 않고 실패 모드를 정상 시나리오로 둔다.
- **민감정보 유출**: live probe나 조사 과정에서 raw token/response/account id가 노출될 수 있다. 저장 금지, redacted summary, mock fixture 원칙을 적용한다.
- **scope creep**: feasibility task가 곧바로 command 구현으로 확장될 수 있다. 채택 결론이 나오면 후속 구현 이슈를 별도로 분리한다.

## 승인 요청 사항

- #30을 feasibility task로 진행하고, 기본 `analyze` 경로와 `UsageSnapshot v2` schema는 변경하지 않는 범위를 승인한다.
- 실제 credential/keychain 접근이나 live remote probe는 구현계획서에서 별도 승인 항목으로 분리하는 방향을 승인한다.
- 산출물은 `mydocs/` 내부 작업 문서에 한정하고, 사용자-facing 문서는 후속 구현 이슈로 미루는 방향을 승인한다.

승인되면 `task_m010_30_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
