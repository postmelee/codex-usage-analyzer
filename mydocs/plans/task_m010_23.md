# Task M010 #23 수행계획서

GitHub Issue: [#23](https://github.com/postmelee/codex-usage-analyzer/issues/23)
마일스톤: M010

## 목적

`codex-usage-analyzer`의 후속 npm release에서 version bump부터 publish 이후 smoke까지 반복 가능한 절차를 문서화한다. 이미 `0.1.0`이 npm에 publish되어 같은 version을 다시 사용할 수 없으므로, 다음 release부터는 semver 판단, `package.json` version 변경, 검증, tag/GitHub Release, trusted publishing workflow 실행, postpublish smoke 순서를 명확히 고정해야 한다.

이번 task는 release 절차 문서화와 automation 분리 필요성 판단에 집중한다. 실제 version bump, git tag 생성, GitHub Release 생성, npm publish 실행은 수행하지 않는다.

## 배경

- #7에서 `codex-usage-analyzer@0.1.0`을 npm에 publish했고 `npx codex-usage-analyzer@latest analyze --json` smoke까지 확인했다.
- #21에서 package license metadata를 MIT로 확정했다.
- #22에서 GitHub Actions OIDC 기반 trusted publishing workflow와 README Trusted Publisher 안내를 추가했다.
- 사용자가 npmjs.com Trusted Publisher 설정을 완료했지만, 실제 publish workflow 실행은 version bump와 release ordering 확정 전까지 보류해야 한다.
- 현재 `package.json` version은 `0.1.0`이고, README `Release Checklist`는 release 전후 검증 명령과 trusted publishing 주의사항은 담고 있지만 semver bump, tag, release note, GitHub Release, publish workflow 실행 순서는 아직 고정하지 않았다.

## 설계 방향

- 기본 방향은 README `Release Checklist`를 반복 가능한 release runbook으로 보강하는 것이다.
- 별도 공식 문서 루트가 없는 현재 저장소 구조를 유지하고, Stage 1에서 README만으로 절차를 담기에 충분한지 확인한다.
- version bump는 `package.json`의 `version` 변경을 중심으로 다루되, npm 공식 `npm version` command 사용 여부와 git tag 생성 동작은 Stage 1에서 확인한 뒤 Stage 2에 반영한다.
- #22의 trusted publishing workflow는 이미 manual-only이므로, #23에서는 workflow를 실행할 release gate와 순서를 문서화한다.
- release note와 PR/issue에는 실제 사용자 snapshot, 로컬 파일 경로, 인증 토큰, 계정 식별자 원본을 기록하지 않는 원칙을 유지한다.
- automation이 문서만으로 충분하지 않다고 판단되면 이번 task에서 구현하지 않고 별도 후속 이슈 후보로 분리한다.

- **Stage 1 — release baseline과 공식 절차 후보 점검**

- **Stage 2 — release ordering 문서 반영**
  - 산출물: README `Release Checklist` 보강, Stage 2 보고서

- **Stage 3 — 통합 검증과 최종 보고**

### 통합 검증

- 후속 release에서 version bump부터 postpublish smoke까지 반복 가능한 절차가 문서화된다.
- `0.1.0` publish 이후 version 재사용 불가 조건과 version bump 필요성이 명확히 기록된다.
- automation을 이번 task에서 하지 않는다면 별도 후속 이슈 후보가 기록된다.
- 문서의 명령어와 절차가 #7의 실제 publish flow 및 #22 trusted publishing workflow와 충돌하지 않는다.
- 이번 task에서 `package.json` version이 변경되지 않는다.
- 이번 task에서 `npm publish`, git tag 생성, GitHub Release 생성이 수행되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **문서 과밀화**: README에 release runbook을 너무 길게 넣으면 사용자가 읽는 기본 문서가 복잡해질 수 있다. Stage 1에서 README 보강 범위를 먼저 판단한다.
- **npm command 부작용**: `npm version`은 설정에 따라 git commit/tag를 만들 수 있으므로, 실제 사용 여부와 명령 예시는 공식 문서 확인 후 보수적으로 기록한다.
- **실수 publish 위험**: trusted publishing workflow는 수동 실행 시 실제 publish를 시도한다. 이번 task에서는 실행하지 않고 문서에 release gate를 명확히 둔다.
- **automation 범위 확장**: version bump automation까지 구현하면 release flow 변경 범위가 커진다. 이번 task에서는 필요성을 판단하고 구현은 별도 이슈로 분리한다.
- **민감 정보 노출**: release note와 smoke 결과 기록 과정에서 raw production snapshot, 로컬 경로, 계정 식별자, credential이 노출될 수 있으므로 기록 가능 범위를 문서에 명시한다.
