# Task M040 #46 수행계획서

GitHub Issue: [#46](https://github.com/postmelee/codex-usage-analyzer/issues/46)
마일스톤: M040

## 목적

기존 기본 명령과 Account Usage Contract v1은 공식 `account/usage/read` 기반 identity-free 경로로 유지하면서, 사용자가 명시적으로 선택했을 때만 Codex Desktop 프로필에 대응하는 추가 정보를 조회하는 `profile` 실험 명령을 구현한다.

사람이 읽는 `profile` 출력에는 프로필, 공식 사용량, 활동 인사이트, 상위 invocation, 일별 token 활동 지도를 제공한다. `profile --json`은 기존 Account Usage Contract를 그대로 중첩하고 실험적 identity/activity 정보를 별도 영역에 담는 버전된 Full Profile Envelope를 반환한다.

## 배경

#30에서 Codex Desktop이 사용하는 비공식 `/wham/profiles/me` 경로와 app-server internal auth method를 분석하고, 작업지시자 승인 아래 일회성 direct-call로 화면 수치와의 parity 및 1초 미만 응답 범주를 확인했다. 이후 기본 CLI는 공식 `account/usage/read` thin wrapper로 재구성되어 현재 인자 없는 실행과 `--json` 모두 인증 material을 package로 가져오지 않는다.

최신 Desktop 추출 번들 분석에서는 profile identity, stats summary, daily usage buckets, activity insights, top invocations의 실제 필드 대응과 token heatmap 계산 방식을 확인했다. 그러나 profile endpoint와 인증 method는 공개 안정 계약이 아니며, 전체 프로필 조회 시 bearer token과 account context가 Node.js process memory를 통과한다. 따라서 이 기능을 기본 경로에 넣으면 현재 README의 identity-free/no-token 보안 약속과 충돌한다.

이번 task는 기본 기능을 변경하지 않고 `profile` 명령 자체를 매 실행의 명시적 opt-in으로 간주한다. 사용자 입력 endpoint, credential file, keychain, raw response 저장을 허용하지 않으며 drift와 인증 실패를 정상적인 실험 기능 실패 모드로 취급한다.

## 설계 방향

- 기본 CLI는 계속 공식 usage 경로만 실행한다. `profile` 명령을 파싱한 경우에만 internal auth와 private profile HTTP adapter를 로드·호출하며 자동 fallback은 두지 않는다.
- stable Account Usage Contract를 확장하지 않는다. Full Profile Envelope는 `fullProfileContractVersion`, 고정 `kind`, `stability: "experimental"`, profile 상태, canonical `usage`, `profile`, `activityInsights`를 가진 별도 계약으로 정의한다.
- envelope의 `usage`는 기존 `readAccountUsage()`와 동일한 normalizer 결과여야 한다. private endpoint의 stats 값은 parity 참고 입력일 뿐 canonical usage를 덮어쓰지 않는다.
- identity allowlist는 `displayName`, `username`, `avatarUrl`, `planType`으로 제한한다. `planType`은 app-server의 account metadata에서 안전하게 얻을 수 있을 때만 사용하고, email과 account/account-user id는 즉시 폐기한다.
- activity allowlist는 `fastModePercent`, `reasoningEffort`, `reasoningEffortPercent`, `skillsExplored`, `totalSkillsUsed`, `totalThreads`, `topInvocations`로 제한한다. invocation은 type, display name, usage count만 정규화하고 plugin/skill 내부 id는 출력하지 않는다.
- root profile 상태는 `ok`, `partial`, `unavailable`로 제한한다. optional field 누락이나 upstream stats error는 `partial`, 인증/endpoint/transport 실패는 `unavailable`로 분류한다. `ok`와 `partial`은 exit 0, `unavailable`은 structured JSON/human summary를 출력한 뒤 exit 1을 사용하는 안을 구현계획서에서 최종 고정한다.
- app-server는 `profile` 실행에서 한 번만 시작해 initialize 이후 공식 usage와 필요한 account/auth context를 수집한다. internal auth method drift는 safe unavailable로 처리하고 raw RPC error message는 버린다.
- Node.js built-in transport만 사용한다. endpoint는 package 내부 상수로 고정하고 사용자 URL 입력, redirect, retry를 금지한다. 응답은 bounded timeout과 byte limit 안에서만 읽고 raw body를 오류에 첨부하지 않는다.
- token/account context는 closure-local 변수로 유지하고 disk, environment, command argument, telemetry, debug output에 복사하지 않는다. JavaScript runtime에서 완전한 memory zeroization을 보장할 수 없다는 한계도 공개 문서에 명시한다.
- `profile --json`의 실험 경고는 stderr에만 출력한다. stdout은 성공, partial, unavailable 상태에서도 문서화된 JSON 한 개만 출력해 downstream parser가 진단 문자열과 섞이지 않게 한다.
- token heatmap은 JSON 계약에 별도 파생 grid를 넣지 않고 canonical daily buckets에서 사람이 읽는 출력 시 계산한다. Desktop과 동일하게 Sunday-start UTC date-only week, 최대 52주, 일별 상대 intensity 0~4를 사용한다.
- 실험 구현은 CLI 내부 모듈로 격리하고 `src/index.js`와 `src/index.d.ts`의 stable SDK surface에는 추가하지 않는다.

## 잠정 단계

- **Stage 1 — Full Profile Envelope와 synthetic normalizer 확정**
  - stable usage를 중첩하는 별도 실험 JSON 계약, field allowlist, status/null semantics를 정의한다.
  - 실제 사용자 payload가 아닌 synthetic fixture로 parser와 JSON Schema 일치를 검증한다.
- **Stage 2 — 격리된 app-server auth와 private profile transport 구현**
  - `profile` 전용 app-server request sequence, process-memory token 경계, 고정 HTTPS adapter를 구현한다.
  - timeout, byte limit, redirect, malformed response, auth/schema drift, cleanup과 민감정보 redaction을 검증한다.
- **Stage 3 — CLI command와 사람이 읽는 profile renderer 구현**
  - `profile`, `profile --json`, help, stderr warning, exit/status 정책을 연결한다.
  - Profile/Usage/Activity insights/Top invocations와 52주 token heatmap 출력 및 기본 usage 회귀를 검증한다.
- **Stage 4 — 공식 문서, package artifact, 통합 보안 검증**
  - README와 공식 contract/downstream 문서를 승인된 안정성 경계에 맞게 갱신한다.
  - 전체 테스트, schema, package dry-run, sensitive pattern scan을 수행하고 실제 계정 smoke는 별도 승인 후 structure/parity boolean만 확인한다.

## 리스크

- **비공식 endpoint와 auth method drift**: current Codex build에서 동작해도 예고 없이 field, method, header 요구가 바뀔 수 있다. 기본 usage와 완전히 분리하고 drift를 `partial` 또는 `unavailable`로 안전하게 처리한다.
- **npm 신뢰 경계 확대**: profile 실행에서는 bearer token이 Node.js memory를 통과한다. 명시적 command, stderr 경고, no persistence, fixed request target, no SDK export로 노출 범위를 제한한다.
- **identity/privacy 노출**: 이름, username, avatar URL은 개인 식별 정보다. 기본 출력과 submit에는 포함하지 않고 사용자가 `profile`을 직접 실행한 terminal/stdout에만 반환한다.
- **SSRF와 credential exfiltration**: endpoint/header 주입이 가능하면 token 유출로 이어질 수 있다. 사용자 지정 URL·redirect·proxy option을 제공하지 않고 고정 HTTPS origin만 허용한다.
- **원본 응답 과수집**: unknown upstream field가 계약에 자동 진입하면 email/id 등 민감값이 섞일 수 있다. 모든 계층에서 allowlist reconstruction을 사용하고 raw object pass-through를 금지한다.
- **stable contract 혼동**: Full Profile Envelope를 Account Usage Contract v2처럼 보이게 하면 downstream이 안정성을 오해할 수 있다. 별도 version key, kind, stability, 파일명, 문서로 분리한다.
- **부분 성공 해석 차이**: usage 성공과 profile 실패가 동시에 존재할 수 있다. status/null/exit code matrix를 Stage 1과 구현계획서에서 고정하고 schema/test로 강제한다.
- **avatar URL 후속 위험**: remote avatar URL은 untrusted input이다. CLI는 문자열 allowlist와 length/scheme 검증만 하고 fetch/re-host하지 않으며 downstream이 안전하게 재처리하도록 문서화한다.
- **runtime memory 한계**: JavaScript는 token buffer의 완전한 zeroization을 보장하지 않는다. 참조 수명 최소화와 process 종료 경계를 적용하고 이 한계를 숨기지 않는다.

## 승인 요청 사항

- 기본 `usage`/`--json`과 stable SDK/Account Usage Contract v1은 변경하지 않고 `profile`만 매 실행 opt-in으로 사용하는 CLI 구조
- Full Profile Envelope를 안정 계약과 분리하고 canonical usage는 공식 `account/usage/read`만 사용하는 방향
- identity/activity allowlist와 plugin/skill id, email, account identifier 제외
- app-server internal auth method와 고정 `/wham/profiles/me` 호출을 profile 전용 process-memory adapter로 허용하는 보안 경계
- 사용자 URL/redirect/retry/auth file/keychain/raw persistence를 금지하는 transport 정책
- `ok`/`partial`은 exit 0, `unavailable`은 structured output 후 exit 1로 두는 잠정 정책을 구현계획서에서 최종 확정하는 방향
- experimental feature를 public JavaScript SDK로 export하지 않는 범위
- `README.md`, `docs/experimental-full-profile.md`, schema, 기존 contract/downstream 문서를 공식 제품 문서로 갱신하는 위치 판단
- 4개 잠정 Stage와 실제 계정 smoke를 별도 승인 gate로 유지하는 검증 범위

승인되면 `task_m040_46_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
