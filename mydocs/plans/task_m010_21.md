# Task M010 #21 수행계획서

GitHub Issue: [#21](https://github.com/postmelee/codex-usage-analyzer/issues/21)
마일스톤: M010

## 목적

`codex-usage-analyzer` npm package와 저장소에 적용할 license metadata 정책을 확정한다. 작업지시자가 승인한 license 값 또는 보류 결정을 기준으로 `package.json`, 루트 license 문서, README license 문구를 일관되게 반영한다.

이번 task는 법률 자문이 아니라 package metadata와 저장소 문서 정합성을 맞추는 작업이다. 에이전트가 MIT, ISC, Apache-2.0 등 특정 license를 임의로 선택하지 않고, 후보와 영향을 정리한 뒤 작업지시자 승인에 따라 반영한다.

## 배경

#7에서 `codex-usage-analyzer@0.1.0` npm publish와 postpublish `npx @latest` smoke를 완료했다. 다만 `package.json`에는 `license` 필드가 없고, 저장소 루트에도 `LICENSE` 파일이 없다.

npm package 사용자는 package metadata와 package contents에서 license 상태를 확인하는 경우가 많다. 현재 상태를 유지할 수도 있지만, 유지한다면 "license metadata를 보류한다"는 결정을 작업 문서에 명확히 남겨야 한다.

## 범위

### 포함

- 현재 `package.json` license metadata 부재 상태 확인
- 루트 `LICENSE` 파일 부재 상태 확인
- license 후보와 반영 범위 정리
- 작업지시자 승인된 license 값 또는 보류 결정 기록
- 승인된 경우 `package.json`의 `license` 필드 추가
- 승인된 경우 루트 `LICENSE` 파일 추가
- 필요한 경우 README license 문구 추가 또는 갱신
- `npm pack --dry-run`으로 package metadata/file 포함 여부 확인

### 제외

- 법률 자문
- 기존 코드 소유권 변경
- contributor license agreement 도입
- npm publish 재실행
- trusted publishing/provenance 도입
- release version bump 절차 문서화
- `UsageSnapshot v2` schema 변경
- analyzer 기능 변경

## 설계 방향

- license 결정은 작업지시자 승인 항목으로 분리한다.
- `package.json`의 `license` 값은 npm/SPDX 관례에 맞는 식별자를 사용한다.
- 루트 `LICENSE` 파일을 추가하는 경우 package contents에 포함되는지 `npm pack --dry-run`으로 확인한다.
- README에는 사용자가 package license 상태를 빠르게 확인할 수 있는 짧은 섹션만 추가한다.
- license metadata를 보류하기로 결정하면 `package.json`/`LICENSE`를 임의 추가하지 않고, 작업 산출물과 README 필요 여부만 재검토한다.
- `files` allowlist가 package contents에 영향을 주므로, 루트 license 파일이 자동 포함되는지 pack 결과로 검증하고 필요할 때만 `files` 조정을 검토한다.

## 잠정 단계

- **Stage 1 — license 결정 정리와 승인 기록**
  - 산출물: 현재 metadata 상태, license 후보, 작업지시자 승인 결정이 포함된 구현 계획서
  - 검증 관점: license를 임의 결정하지 않고 승인된 값 또는 보류 결정을 명확히 기록했는지 확인
- **Stage 2 — package/document 반영**
  - 산출물: 승인 결정에 따른 `package.json`, `LICENSE`, README 변경
  - 검증 관점: metadata, license 문서, README 문구가 서로 충돌하지 않는지 확인
- **Stage 3 — package contents 검증과 최종 보고**
  - 산출물: `npm pack --dry-run` 결과 확인, 최종 보고서, PR 준비 상태
  - 검증 관점: package contents와 metadata가 의도한 license 상태를 나타내는지 확인

## 검증 계획

### 단계별 검증

- Stage 1
  - `git status --short`
  - license 결정 또는 보류 결정이 구현 계획서에 기록됐는지 수동 확인
- Stage 2
  - `node -e "JSON.parse(require('fs').readFileSync('package.json', 'utf8')); console.log('package.json ok')"`
  - README/license 문구 수동 확인
- Stage 3
  - `npm test`
  - `npm pack --dry-run`
  - `git diff --check`

### 통합 검증

- `package.json`, `LICENSE`, README가 승인된 license 결정과 일치한다.
- license metadata 보류 결정인 경우 임의 license 파일이나 metadata가 추가되지 않는다.
- `npm pack --dry-run` 결과에서 license 관련 metadata/file 포함 여부가 의도와 일치한다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **license 임의 결정 위험**: license는 법적/정책적 의미가 있으므로 작업지시자 승인 전에는 package metadata에 추가하지 않는다.
- **package contents 오판**: `files` allowlist와 npm의 always-included file 규칙이 섞일 수 있으므로 `npm pack --dry-run` 결과로 실제 포함 여부를 확인한다.
- **문서 불일치**: `package.json`, `LICENSE`, README가 서로 다른 license 상태를 말하지 않도록 Stage 2에서 함께 검토한다.
- **published package와 저장소 불일치**: 이번 task는 npm publish를 하지 않으므로 npm registry의 `0.1.0` metadata는 즉시 바뀌지 않는다. 이 제한을 최종 보고에 명시한다.

## 승인 요청 사항

- license 결정 방식 승인: Stage 1에서 후보를 정리한 뒤 작업지시자가 최종 license 값 또는 metadata 보류를 선택
- 공식 문서 위치 판단 승인: 루트 `LICENSE`, 루트 `README.md`, 루트 `package.json`만 사용자-facing 문서로 사용
- 이번 task에서 npm publish, trusted publishing/provenance, version bump 절차 문서화를 제외하는 범위 승인
