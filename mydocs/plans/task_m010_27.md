# Task M010 #27 수행계획서

GitHub Issue: [#27](https://github.com/postmelee/codex-usage-analyzer/issues/27)
마일스톤: M010

## 목적

`codex-usage-analyzer`의 다음 npm release 전에 README `Release Checklist`를 사람이 빠뜨리지 않았는지 확인할 read-only preflight/checklist script를 추가한다. 이 script는 release 준비 상태를 관찰하고 실패 원인을 설명하되, version bump, tag 생성, npm publish, GitHub Release 생성 같은 mutation은 수행하지 않는다.

이번 task는 실제 release 수행이 아니라 release 전 검증 도구를 만드는 작업이다. 현재 `package.json` version과 npm latest가 모두 `0.1.0`인 상태에서도 작업 검증이 가능해야 하므로, version 비교 실패/경고 기준과 release-ready 모드 필요 여부를 Stage 1에서 먼저 확정한다.

## 배경

- #7에서 `codex-usage-analyzer@0.1.0`을 npm에 publish했고 postpublish `npx @latest` smoke를 확인했다.
- #22에서 GitHub Actions trusted publishing workflow를 추가했고 npmjs.com Trusted Publisher 설정도 완료했다.
- #23에서 version bump부터 publish 이후 검증까지 release ordering을 README에 문서화했다.
- 다음 실제 release 전에 사람이 checklist를 빠뜨리지 않았는지 확인할 read-only preflight script가 있으면 npm publish 실수 위험을 줄일 수 있다.
- 현재 `package.json`에는 `test` script만 있고 release preflight entrypoint는 없다.
- 현재 `scripts/`에는 `profile-smoke.js`만 있다.

## 설계 방향

- Node.js ESM script를 추가하고 외부 runtime dependency는 추가하지 않는다.
- script는 `node:child_process`, `node:fs`, `node:path` 표준 API 중심으로 read-only command를 orchestration한다.
- mutation 가능성이 있는 명령은 script에 넣지 않는다. 금지 예: `npm version`, `npm publish`, `git tag`, `git push`, `gh release create`.
- `npm view codex-usage-analyzer version dist-tags --json`은 registry 상태 확인용 read-only network query로 취급한다.
- `npm pack --dry-run`은 package contents 확인용으로 사용하고 tarball 부산물이 남지 않는지 검증한다.
- 현재 repo는 아직 다음 version bump를 하지 않았으므로, Stage 1에서 다음 중 하나를 결정한다.
  - 기본 `npm run release:preflight`는 구조 점검은 통과하되 version 재사용 위험을 경고로 표시하고, release 직전에는 strict flag로 실패 처리한다.
  - 기본 `npm run release:preflight`가 version 재사용 위험을 실패 처리하고, 이번 task 검증은 예상 실패 출력까지 수용 기준으로 기록한다.
- 실패 시에는 어떤 check가 실패했는지 사람이 조치할 수 있게 짧은 요약과 exit code를 제공한다.
- README `Release Checklist`에는 version bump PR에서 preflight를 언제 실행해야 하는지와 strict/normal 모드가 있다면 그 차이를 명시한다.

- preflight 실패 또는 경고 시 어떤 조건을 조치해야 하는지 출력한다.
- script가 `package.json` version, npm latest, tag 상태, publish workflow 조건을 확인한다.
- script 실행만으로 파일, tag, npm registry, GitHub Release 상태가 변경되지 않는다.
- README release checklist에 preflight 실행 시점이 문서화된다.
- 이번 task에서 version bump, git tag 생성, npm publish, GitHub Release 생성이 수행되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **현재 version과 npm latest 동일**: 아직 다음 release prep 전이라 strict version check는 현재 상태에서 실패할 수 있다. Stage 1에서 기본 모드와 strict/release-ready 모드를 분리할지 결정한다.
- **read-only 경계**: preflight script가 실수로 mutation command를 실행하면 release 안전 도구가 오히려 위험해진다. command allowlist와 source review를 검증에 포함한다.
- **network 의존성**: npm latest 조회는 registry network가 필요하다. 실패 시 메시지를 명확히 하고, offline fallback을 둘지 Stage 1에서 판단한다.
- **중복 검증 비용**: preflight가 `npm test`와 `npm pack --dry-run`을 실행하면 시간이 늘어난다. release 전 안전성을 우선하되 출력이 과도하지 않게 구성한다.
- **README 과밀화**: release checklist가 길어질 수 있으므로 README에는 실행 시점과 핵심 명령만 보강한다.
